# Advanced JavaScript Questions & Answers: 401–500

---

## 401. What is the difference between unit, integration, and end-to-end tests?

Unit tests check small isolated pieces. Integration tests check multiple parts working together. End-to-end tests simulate real user flows.

```js id="2syw76"
function add(a, b) {
  return a + b;
}

// Unit test
console.assert(add(2, 3) === 5);
```

A senior answer explains that confidence increases from unit to E2E, but speed and debuggability usually decrease.

---

## 402. What makes a function easy to test?

Pure functions with explicit inputs and outputs are easiest to test.

```js id="e3kb17"
function calculateTotal(items) {
  return items.reduce((sum, item) => sum + item.price, 0);
}

console.assert(
  calculateTotal([{ price: 10 }, { price: 20 }]) === 30
);
```

No network, time, randomness, DOM, or global state is involved.

---

## 403. Why is global state hard to test?

Global state makes tests affect each other.

```js id="22x69u"
let count = 0;

function increment() {
  count++;
  return count;
}

console.log(increment()); // 1
console.log(increment()); // 2
```

Tests must reset global state or they become order-dependent.

---

## 404. What is dependency injection?

Dependency injection means passing dependencies in instead of importing or creating them directly.

```js id="kk6cat"
function createUserService({ db, logger }) {
  return {
    async createUser(user) {
      logger.info("Creating user");
      return db.insert(user);
    }
  };
}
```

This makes testing easier because `db` and `logger` can be mocked.

---

## 405. What is mocking?

Mocking replaces a real dependency with a fake one for testing.

```js id="yojx2q"
const mockDb = {
  insert(user) {
    return Promise.resolve({ id: 1, ...user });
  }
};

const service = createUserService({
  db: mockDb,
  logger: { info() {} }
});
```

Mocks are useful, but over-mocking can test implementation details instead of behavior.

---

## 406. What is spying in tests?

A spy records how a function was called.

```js id="cc7m8k"
function createSpy() {
  const calls = [];

  function spy(...args) {
    calls.push(args);
  }

  spy.calls = calls;
  return spy;
}

const log = createSpy();

log("hello");
log("world");

console.log(log.calls); // [["hello"], ["world"]]
```

Spies help verify side effects.

## 407. What is a flaky test?

A flaky test sometimes passes and sometimes fails without code changes.

```js id="fnwdfn"
test("bad timing test", done => {
  setTimeout(() => {
    expect(value).toBe(1);
    done();
  }, 100);
});
```

Flakiness often comes from time, randomness, network, shared state, or race conditions.

---

## 408. Why are fixed sleeps bad in tests?

Fixed sleeps make tests slow and unreliable.

```js id="8lm3cs"
// Bad
await new Promise(resolve => setTimeout(resolve, 1000));

expect(screen.textContent).toContain("Loaded");
```

Better: wait for the actual condition.

```js id="dyrjqi"
async function waitFor(condition, timeout = 1000) {
  const start = Date.now();

  while (Date.now() - start < timeout) {
    if (condition()) return;
    await new Promise(resolve => setTimeout(resolve, 10));
  }

  throw new Error("Timed out");
}
```

## 409. What is fake time?

Fake time lets tests control timers.

```js id="yq560h"
function debounce(fn, delay) {
  let id;

  return (...args) => {
    clearTimeout(id);
    id = setTimeout(() => fn(...args), delay);
  };
}
```

In real test frameworks, fake timers allow advancing time without waiting in real life.

---

## 410. Why do fake timers and promises interact strangely?

Timers use task queues. Promises use microtasks. Advancing timers may not automatically flush pending promise callbacks.

```js id="b42grm"
setTimeout(() => {
  Promise.resolve().then(() => {
    console.log("promise inside timer");
  });
}, 0);
```

A robust test may need to advance timers and flush microtasks.

## 411. What is snapshot testing?

Snapshot testing compares output to a saved expected result.

```js id="8fmje3"
function renderUser(user) {
  return `<p>${user.name}</p>`;
}

const output = renderUser({ name: "Ava" });

console.log(output); // compare with stored snapshot
```

Snapshots are useful for broad changes but can become noisy if reviewed carelessly.

## 412. What is property-based testing?

Property-based testing checks general rules over many generated inputs.

```js id="frfcn5"
function reverseTwice(arr) {
  return [...arr].reverse().reverse();
}

const input = [1, 2, 3];

console.assert(
  JSON.stringify(reverseTwice(input)) === JSON.stringify(input)
);
```

The property is: reversing twice should produce the original array.

## 413. What is mutation testing?

Mutation testing changes your code slightly to see whether tests catch the change.

```js id="4dtgrg"
// Original
function isAdult(age) {
  return age >= 18;
}

// Mutated
function isAdult(age) {
  return age > 18;
}
```

If tests still pass, they may be weak.

## 414. What is test isolation?

Each test should run independently.

```js id="tlf4va"
let users = [];

function beforeEach() {
  users = [];
}

function testCreateUser() {
  beforeEach();
  users.push({ name: "Ava" });
  console.assert(users.length === 1);
}
```

Shared mutable state breaks isolation.

## 415. What is the Arrange-Act-Assert pattern?

It structures tests into setup, execution, and verification.

```js id="2xv7ze"
// Arrange
const cart = [{ price: 10 }, { price: 20 }];

// Act
const total = calculateTotal(cart);

// Assert
console.assert(total === 30);
```

This keeps tests readable.

## 416. What is the difference between error and exception?

In JavaScript, an exception is a thrown value. Usually that value is an `Error` object.

```js id="5m0vyn"
throw new Error("Something failed");
```

Technically, JavaScript can throw anything:

```js id="wdo18k"
throw "bad";
```

But throwing `Error` objects is better because they contain stack traces.

---

## 417. Why should you not throw strings?

Strings do not provide useful stack or structured error information.

```js id="ke1jqz"
// Bad
throw "User not found";
```

Better:

```js id="11uxjr"
throw new Error("User not found");
```

Custom errors are even better for domain-specific handling.

## 418. What is a custom error class?

A custom error class represents a specific failure type.

```js id="nd47hr"
class ValidationError extends Error {
  constructor(message, details) {
    super(message);
    this.name = "ValidationError";
    this.details = details;
  }
}

throw new ValidationError("Invalid email", {
  field: "email"
});
```

This allows precise error handling.

## 419. What is error cause?

`cause` preserves the original error when wrapping it.

```js id="0qx3ho"
try {
  JSON.parse("{bad json}");
} catch (error) {
  throw new Error("Failed to parse config", {
    cause: error
  });
}
```

This keeps higher-level context without losing the original failure.

---

## 420. What is operational error vs programmer error?

Operational errors are expected runtime failures, like network errors. Programmer errors are bugs.

```js id="c1ay5e"
async function loadUser(id) {
  const res = await fetch(`/users/${id}`);

  if (!res.ok) {
    throw new Error("Failed to load user");
  }

  return res.json();
}
```

A failed request is operational. Accessing a property of `undefined` is usually programmer error.

## 421. What is fail-fast?

Fail-fast means detecting invalid conditions early.

```js id="503eqq"
function transfer(amount) {
  if (amount <= 0) {
    throw new Error("Amount must be positive");
  }

  // continue
}
```

This prevents bad state from spreading deeper into the system.

---

## 422. What is defensive programming?

Defensive programming checks assumptions at boundaries.

```js id="tmtsb4"
function parseUser(input) {
  if (!input || typeof input.name !== "string") {
    throw new Error("Invalid user");
  }

  return {
    name: input.name
  };
}
```

Use it especially when receiving external input.

---

## 423. What is input validation?

Input validation ensures data matches expected shape and constraints.

```js id="6virdu"
function validateEmail(value) {
  if (typeof value !== "string") return false;
  return value.includes("@");
}

console.log(validateEmail("a@example.com")); // true
```

Validation should happen at system boundaries: APIs, forms, messages, files, and environment variables.

---

## 424. What is schema validation?

Schema validation checks data against a formal structure.

```js id="7b4r3s"
function validateUser(value) {
  return (
    value &&
    typeof value.id === "number" &&
    typeof value.name === "string"
  );
}
```

In real projects, libraries often define schemas for runtime checks.

## 425. Why is TypeScript not enough for runtime validation?

TypeScript types disappear at runtime.

```ts id="tf7qtm"
type User = {
  id: number;
  name: string;
};

const user = JSON.parse('{"id":"wrong","name":123}') as User;
```

The compiler trusts the assertion, but runtime data is still invalid.

---

## 426. What is type erasure?

Type erasure means TypeScript types are removed during compilation.

```ts id="t2pru9"
function greet(name: string) {
  return `Hello ${name}`;
}
```

Compiles roughly to:

```js id="kg8q77"
function greet(name) {
  return `Hello ${name}`;
}
```

No runtime type check remains.

## 427. What is structural typing?

TypeScript uses structural typing: compatibility is based on shape, not explicit declaration.

```ts id="q46k18"
type User = {
  name: string;
};

const obj = {
  name: "Ava",
  age: 30
};

const user: User = obj;
```

Because `obj` has `name`, it is compatible.

---

## 428. What is nominal typing?

Nominal typing means types are compatible only if explicitly declared as the same type.

JavaScript does not have nominal static types, but runtime branding can approximate it.

```js id="tldw6q"
const UserBrand = Symbol("UserBrand");

function createUser(name) {
  return {
    [UserBrand]: true,
    name
  };
}

function isUser(value) {
  return Boolean(value?.[UserBrand]);
}
```

---

## 429. What is branding?

Branding adds a hidden marker to distinguish otherwise similar values.

```js id="t5jvea"
const EmailBrand = Symbol("Email");

function createEmail(value) {
  if (!value.includes("@")) {
    throw new Error("Invalid email");
  }

  return {
    value,
    [EmailBrand]: true
  };
}
```

Useful when two strings have different meanings.

---

## 430. What is a discriminated union conceptually?

A discriminated union uses a shared tag field to distinguish variants.

```js id="fkiq4f"
function handleResult(result) {
  switch (result.type) {
    case "success":
      return result.value;

    case "error":
      throw new Error(result.message);
  }
}

handleResult({
  type: "success",
  value: 42
});
```

This pattern works in plain JavaScript too.

---

## 431. What is exhaustive checking conceptually?

Exhaustive checking ensures every variant is handled.

```js id="06v9bc"
function assertNever(value) {
  throw new Error(`Unhandled case: ${JSON.stringify(value)}`);
}

function handle(status) {
  switch (status.kind) {
    case "loading":
      return "Loading";

    case "success":
      return "Done";

    case "error":
      return "Failed";

    default:
      return assertNever(status);
  }
}
```

In TypeScript, this can be checked statically. In JS, it fails at runtime.

## 432. What is duck typing?

Duck typing means using an object based on what it can do, not what class it belongs to.

```js id="da4jjm"
function play(soundMaker) {
  soundMaker.makeSound();
}

play({
  makeSound() {
    console.log("beep");
  }
});
```

“If it quacks like a duck, treat it like a duck.”

## 433. What is feature testing in objects?

Check whether a method or property exists before using it.

```js id="aquz99"
function canSerialize(value) {
  return typeof value?.toJSON === "function";
}
```

This is common in flexible APIs.

## 434. What is an assertion function?

An assertion function throws if a condition is not met.

```js id="rg2z8a"
function assert(condition, message = "Assertion failed") {
  if (!condition) {
    throw new Error(message);
  }
}

function divide(a, b) {
  assert(b !== 0, "Cannot divide by zero");
  return a / b;
}
```

Assertions help document invariants.

## 435. What is an invariant?

An invariant is something that must always be true.

```js id="ph2qml"
class Stack {
  #items = [];

  pop() {
    if (this.#items.length === 0) {
      throw new Error("Cannot pop empty stack");
    }

    return this.#items.pop();
  }
}
```

The invariant is that `pop` must not return from an empty stack.

## 436. What is an API boundary?

An API boundary is where data crosses from one system or layer to another.

```js id="4sauzl"
async function getUserFromApi(id) {
  const response = await fetch(`/api/users/${id}`);
  const data = await response.json();

  return parseUser(data);
}
```

Validate and normalize data at boundaries.

## 437. What is data normalization?

Normalization converts data into a predictable shape.

```js id="8w1co7"
function normalizeUser(raw) {
  return {
    id: Number(raw.id),
    name: String(raw.name).trim(),
    active: Boolean(raw.active)
  };
}
```

This reduces conditional logic throughout the app.

## 438. What is data denormalization?

Denormalization duplicates or reshapes data for convenient reading.

```js id="cry2e0"
const user = {
  id: 1,
  name: "Ava"
};

const post = {
  id: 10,
  title: "Hello",
  authorName: user.name
};
```

It can improve read performance but risks stale duplicated data.

## 439. What is entity normalization in client state?

Store entities by ID instead of nesting repeated objects.

```js id="8ampwv"
const state = {
  users: {
    byId: {
      1: { id: 1, name: "Ava" }
    },
    allIds: [1]
  }

```

This avoids duplication and makes updates easier.

## 440. What is derived state?

Derived state is computed from existing state.

```js id="0b7f6j"
const cart = [
  { price: 10, quantity: 2 },
  { price: 5, quantity: 1 }
];

const total = cart.reduce(
  (sum, item) => sum + item.price * item.quantity,
  0
);
```

Do not store derived state unless caching is necessary.

## 441. Why can duplicated state cause bugs?

If the same fact is stored in multiple places, one copy can become stale.

```js id="r3jipl"
const state = {
  items: [{ price: 10 }, { price: 20 }],
  total: 30
};

state.items.push({ price: 5 });
// total is now stale
```

Better derive `total` when needed.

## 442. What is command-query separation?

A function should either perform an action or return data, not both, when possible.

```js id="aeynb6"
// Query
function getTotal(cart) {
  return cart.reduce((sum, item) => sum + item.price, 0);
}

// Command
function addItem(cart, item) {
  cart.push(item);
}
```

This makes behavior easier to reason about.

## 443. What is CQRS conceptually?

CQRS separates read models from write models.

```js id="na3bqw"
const writeModel = {
  createOrder(order) {
    // validate and save
  }
};

const readModel = {
  getOrderSummary(id) {
    // optimized for display
  }
};
```

This is useful when read and write requirements differ greatly.

## 444. What is event sourcing conceptually?

Event sourcing stores events instead of only current state.

```js id="cvfz78"
const events = [
  { type: "deposit", amount: 100 },
  { type: "withdraw", amount: 30 }
];

function getBalance(events) {
  return events.reduce((balance, event) => {
    if (event.type === "deposit") return balance + event.amount;
    if (event.type === "withdraw") return balance - event.amount;
    return balance;
  }, 0);
}

console.log(getBalance(events)); // 70
```

State is reconstructed from history.

## 445. What is a reducer?

A reducer takes current state and an action, then returns next state.

```js id="8pkijr"
function counterReducer(state, action) {
  switch (action.type) {
    case "increment":
      return state + 1;

    case "decrement":
      return state - 1;

    default:
      return state;
  }
}
```

Reducers are predictable when they are pure.

---

## 446. Why should reducers be pure?

Pure reducers are easy to test, replay, and debug.

```js id="d8mgav"
// Bad reducer
function reducer(state, action) {
  localStorage.setItem("state", JSON.stringify(state));
  return state;
}
```

Better:

```js id="nvx3s7"
function reducer(state, action) {
  if (action.type === "setName") {
    return {
      ...state,
      name: action.name
    };
  }

  return state;
}
```

Side effects should happen outside the reducer.

## 447. What is middleware conceptually?

Middleware intercepts operations to add behavior.

```js id="743ass"
function withLogging(handler) {
  return async function (request) {
    console.log("Request", request.url);

    const response = await handler(request);

    console.log("Response", response.status);

    return response;
  };
}
```

Middleware is common in servers, state managers, and fetch wrappers.

## 448. What is a higher-order function?

A higher-order function takes a function or returns a function.

```js id="mat05v"
function once(fn) {
  let called = false;
  let result;

  return function (...args) {
    if (!called) {
      called = true;
      result = fn.apply(this, args);
    }

    return result;
  };
}
```

Closures make higher-order functions powerful.

## 449. What is a higher-order component conceptually?

A higher-order component wraps a component to add behavior.

```js id="zlfal1"
function withAuth(renderComponent) {
  return function renderWithAuth(props) {
    if (!props.user) {
      return "Please log in";
    }

    return renderComponent(props);
  };
}
```

The idea is function composition applied to UI components.

## 450. What is the render props pattern conceptually?

A render prop passes rendering logic as a function.

function DataLoader({ render }) {
  const data = { name: "Ava" };

  return render(data);
}

DataLoader({
  render(data) {
    return `<p>${data.name}</p>`;
  }

It allows behavior reuse while letting callers control output.

## 451. What is inversion of control?

Inversion of control means your code gives control to a framework or callback.

```js id="vxm3bz"
button.addEventListener("click", () => {
  console.log("Browser calls this later");
});
```

You register behavior; the host decides when to call it.

## 452. What is callback hell?

Callback hell is deeply nested asynchronous code.

```js id="gpg5f2"
getUser(id, user => {
  getPosts(user.id, posts => {
    getComments(posts[0].id, comments => {
      console.log(comments);
    });
  });
});
```

Promises and async/await flatten this structure.

```js id="ru7swy"
const user = await getUser(id);
const posts = await getPosts(user.id);
const comments = await getComments(posts[0].id);
```

## 453. What is error-first callback style?

Node.js commonly uses callbacks where the first argument is an error.

```js id="e5he1k"
fs.readFile("file.txt", "utf8", (error, data) => {
  if (error) {
    console.error(error);
    return;
  }

  console.log(data);
});
```

This convention predates widespread promises.

## 454. What is promisification?

Promisification converts callback-based APIs into promise-based APIs.

```js id="ka9ucn"
function readFilePromise(path) {
  return new Promise((resolve, reject) => {
    fs.readFile(path, "utf8", (error, data) => {
      if (error) reject(error);
      else resolve(data);
    });
  });
}
```

Node also provides `util.promisify`.

## 455. What is callback Zalgo?

Zalgo happens when a callback sometimes runs synchronously and sometimes asynchronously.

```js id="kt1ytk"
function getCachedData(key, callback) {
  if (cache.has(key)) {
    callback(cache.get(key)); // sync
  } else {
    fetchData(key).then(callback); // async
  }
}
```

Better: always async.

```js id="pkgshr"
function getCachedData(key, callback) {
  if (cache.has(key)) {
    queueMicrotask(() => callback(cache.get(key)));
  } else {
    fetchData(key).then(callback);
  }
}
```

## 456. Why is consistent async behavior important?

Inconsistent timing causes hard-to-debug ordering bugs.

```js id="tviv3x"
let ready = false;

getCachedData("user", () => {
  console.log(ready);
});

ready = true;
```

If callback runs synchronously, it logs `false`. If async, it logs `true`.

## 457. What is a continuation?

A continuation is the rest of the computation represented as a function.

```js id="5cy01f"
function addAsync(a, b, continuation) {
  continuation(a + b);
}

addAsync(2, 3, result => {
  console.log("Continue with", result);
});
```

Promise callbacks and async resumptions are forms of continuation.

## 458. What is continuation-passing style?

Continuation-passing style passes the next step as a function.

```js id="dr0rsx"
function multiply(a, b, next) {
  next(a * b);
}

multiply(2, 3, result => {
  console.log(result);

```

It makes control flow explicit but can become nested.

## 459. What is a trampoline?

A trampoline turns recursion into iteration to avoid stack overflow.

```js id="9yxu8m"
function trampoline(fn) {
  while (typeof fn === "function") {
    fn = fn();
  }

  return fn;
}

function sum(n, acc = 0) {
  if (n === 0) return acc;

  return () => sum(n - 1, acc + n);
}

console.log(trampoline(() => sum(100000))); 
```

Instead of deep recursive calls, each step returns the next function.

## 460. What is CPS useful for?

CPS can model async flow, early exits, custom control flow, and interpreters.

```js id="t5sga6"
function divide(a, b, onSuccess, onError) {
  if (b === 0) {
    onError(new Error("Divide by zero"));
    return;
  }

  onSuccess(a / b);
}
```

Promises are often easier, but CPS explains many async patterns.

## 461. What is recursion vs iteration?

Recursion solves a problem by calling itself. Iteration uses loops.

```js id="u0gyfh"
function factorialRecursive(n) {
  if (n <= 1) return 1;
  return n * factorialRecursive(n - 1);
}

function factorialIterative(n) {
  let result = 1;

  for (let i = 2; i <= n; i++) {
    result *= i;
  }

  return result;
}
```

JavaScript recursion can hit stack limits.

---

## 462. What is mutual recursion?

Mutual recursion happens when functions call each other.

```js id="5rjtqj"
function isEven(n) {
  if (n === 0) return true;
  return isOdd(n - 1);
}

function isOdd(n) {
  if (n === 0) return false;
  return isEven(n - 1);
}
```

Large inputs can still overflow the stack.

## 463. What is a finite state machine?

A finite state machine models behavior as states and transitions.

```js id="y3cwy5"
const machine = {
  idle: {
    START: "loading"
  },
  loading: {
    SUCCESS: "success",
    FAILURE: "error"
  },
  success: {},
  error: {
    RETRY: "loading"
  }
};

function transition(state, event) {
  return machine[state][event] ?? state;
}

console.log(transition("idle", "START")); // "loading"
```

Useful for UI flows and async states.

## 464. Why are explicit states better than booleans?

Multiple booleans can create impossible states.

```js id="ffy33w"
const state = {
  isLoading: true,
  isSuccess: true,
  isError: false
};
```

Better:

```js id="bszbzv"
const state = {
  status: "loading"
};
```

A single state field prevents invalid combinations.

## 465. What is a state explosion?

State explosion happens when many independent flags create too many combinations.

```js id="hwmqpr"
const state = {
  loading: false,
  saving: false,
  editing: false,
  error: false
};
```

Four booleans create sixteen possible combinations. Many may be invalid.

## 466. What is a reducer-based state machine?

It uses a reducer to enforce transitions.

```js id="769dqd"
function reducer(state, event) {
  switch (state.status) {
    case "idle":
      if (event.type === "START") {
        return { status: "loading" };
      }
      return state;

    case "loading":
      if (event.type === "SUCCESS") {
        return { status: "success", data: event.data };
      }
      if (event.type === "FAILURE") {
        return { status: "error", error: event.error };
      }
      return state;

    default:
      return state;
  }
}
```

This keeps transitions centralized.

---

## 467. What is eventual consistency in frontend apps?

The UI may temporarily differ from server truth.

```js id="0zja02"
post.liked = true;

fetch(`/posts/${post.id}/like`, {
  method: "POST"
}).catch(() => {
  post.liked = false;
});
```

Optimistic updates create temporary inconsistency until server confirmation.

---

## 468. What is conflict resolution?

Conflict resolution decides what happens when two updates disagree.

```js id="t9bcj7"
const local = {
  name: "Ava",
  updatedAt: 200
};

const remote = {
  name: "Mia",
  updatedAt: 250
};

const winner = remote.updatedAt > local.updatedAt ? remote : local;
```

Simple rule: last-write-wins. More complex systems use merging or user choice.

---

## 469. What is last-write-wins?

The newest update overwrites older updates.

```js id="shyuvd"
function merge(a, b) {
  return a.updatedAt > b.updatedAt ? a : b;
}
```

It is simple but can lose data.

---

## 470. What is a CRDT conceptually?

A CRDT is a data structure designed to merge concurrent changes without conflicts.

```js id="mmbjvk"
function mergeCounters(a, b) {
  return {
    clientA: Math.max(a.clientA ?? 0, b.clientA ?? 0),
    clientB: Math.max(a.clientB ?? 0, b.clientB ?? 0)
  };
}
```

The final state converges regardless of update order.

---

## 471. What is immutability with nested updates?

You copy each changed level.

```js id="k1euev"
const state = {
  user: {
    profile: {
      name: "Ava"
    }
  }
};

const next = {
  ...state,
  user: {
    ...state.user,
    profile: {
      ...state.user.profile,
      name: "Mia"
    }
  }
};
```

Only changed paths get new references.

---

## 472. Why is deep cloning often the wrong solution?

Deep cloning copies everything, even unchanged data.

```js id="shp52d"
const next = structuredClone(state);
next.user.profile.name = "Mia";
```

This is simple but may be wasteful and breaks referential sharing.

---

## 473. What is copy-on-write?

Copy-on-write copies data only when modification happens.

```js id="yhrmgk"
function updateName(state, name) {
  return {
    ...state,
    user: {
      ...state.user,
      name
    }
  };
}
```

Unchanged branches are reused.

---

## 474. What is persistent data structure conceptually?

A persistent data structure preserves previous versions while sharing structure.

```js id="x66bkr"
const version1 = {
  user: { name: "Ava" },
  settings: { theme: "dark" }
};

const version2 = {
  ...version1,
  user: {
    ...version1.user,
    name: "Mia"
  }
};

console.log(version1.settings === version2.settings); // true
```

Old and new versions coexist efficiently.

---

## 475. What is a selector?

A selector extracts or derives data from state.

```js id="rp6asf"
function selectCartTotal(state) {
  return state.cart.items.reduce(
    (sum, item) => sum + item.price * item.quantity,
    0
  );
}
```

Selectors centralize data access logic.

---

## 476. What is memoized selector?

A memoized selector caches derived results.

```js id="gu8il0"
function createTotalSelector() {
  let lastItems;
  let lastResult;

  return function selectTotal(state) {
    if (state.cart.items === lastItems) {
      return lastResult;
    }

    lastItems = state.cart.items;
    lastResult = state.cart.items.reduce(
      (sum, item) => sum + item.price,
      0
    );

    return lastResult;
  };
}
```

Works best with immutable state.

---

## 477. Why does memoization depend on referential equality?

Memoization often checks whether references are the same.

```js id="gcv8yj"
const a = [1, 2, 3];
const b = [1, 2, 3];

console.log(a === b); // false
```

Even equal-looking arrays are different references.

---

## 478. What is stale closure in UI code?

A stale closure captures an old value.

```js id="gyxv8t"
function createLogger(count) {
  return function log() {
    console.log(count);
  };
}

const log = createLogger(0);

// Later count changes elsewhere, but this closure still has 0.
log(); // 0
```

Closures capture bindings from their creation context.

---

## 479. How do you avoid stale closure bugs?

Read the latest value from a mutable reference or pass the value explicitly.

```js id="5y9zyz"
const stateRef = {
  current: 0
};

function logLatest() {
  console.log(stateRef.current);
}

stateRef.current = 5;
logLatest(); // 5
```

The closure captures the ref object, not a stale primitive value.

---

## 480. What is a stale async response?

An older request finishes after a newer one and overwrites fresh data.

```js id="v28yrr"
let currentRequest = 0;

async function search(query) {
  const requestId = ++currentRequest;

  const result = await fetch(`/search?q=${query}`).then(r => r.json());

  if (requestId !== currentRequest) return;

  render(result);
}
```

Request IDs or cancellation prevent stale updates.

---

## 481. What is debounced async search?

It delays API calls until the user stops typing.

```js id="a1xyw8"
function debounceAsync(fn, delay) {
  let timerId;

  return function (...args) {
    clearTimeout(timerId);

    return new Promise(resolve => {
      timerId = setTimeout(() => {
        resolve(fn(...args));
      }, delay);
    });
  };
}
```

Combine with cancellation or request IDs for correctness.

---

## 482. What is a cancellable abstraction?

It exposes a way to stop work.

```js id="1580or"
function startTask() {
  const controller = new AbortController();

  const promise = fetch("/data", {
    signal: controller.signal
  });

  return {
    promise,
    cancel() {
      controller.abort();
    }
  };
}

const task = startTask();
task.cancel();
```

Cancellation is essential for UI navigation and cleanup.

---

## 483. What is teardown logic?

Teardown releases resources created during setup.

```js id="a2h4bz"
function setup() {
  const id = setInterval(() => {
    console.log("tick");
  }, 1000);

  return function teardown() {
    clearInterval(id);
  };
}

const cleanup = setup();
cleanup();
```

Timers, listeners, observers, subscriptions, and sockets need teardown.

---

## 484. What is a subscription?

A subscription receives updates over time and usually returns an unsubscribe function.

```js id="bjz4ea"
function subscribe(listener) {
  listeners.add(listener);

  return function unsubscribe() {
    listeners.delete(listener);
  };
}
```

Always unsubscribe when no longer needed.

---

## 485. What is a memory leak from subscriptions?

If the subscriber is never removed, its closure can stay alive.

```js id="ul9ncp"
const listeners = new Set();

function mountComponent() {
  const largeState = new Array(100000).fill("x");

  function onUpdate() {
    console.log(largeState.length);
  }

  listeners.add(onUpdate);
}

mountComponent();
```

`largeState` remains reachable through `listeners`.

---

## 486. What is a weak subscription pattern?

Weak references can avoid retaining listeners strongly, but they are advanced and not always appropriate.

```js id="z782lk"
const listeners = new Set();

function subscribe(obj) {
  listeners.add(new WeakRef(obj));
}
```

Weak references should not replace explicit cleanup in most application code.

---

## 487. What is `WeakRef`?

`WeakRef` holds a weak reference to an object.

```js id="fb0ltc"
let obj = { name: "Ava" };

const ref = new WeakRef(obj);

obj = null;

const value = ref.deref();

if (value) {
  console.log(value.name);
}
```

The object may or may not have been collected when `deref()` is called.

---

## 488. What is `FinalizationRegistry`?

It lets you register cleanup callbacks after an object is garbage collected.

```js id="01w6b7"
const registry = new FinalizationRegistry(value => {
  console.log("Collected:", value);
});

let obj = {};

registry.register(obj, "my object");

obj = null;
```

Do not rely on it for critical cleanup because GC timing is nondeterministic.

---

## 489. Why is GC nondeterministic?

Garbage collection runs when the engine decides it is useful, not immediately when an object becomes unreachable.

```js id="1um7p3"
let obj = { data: "large" };

obj = null;

// Object is eligible for GC,
// but it may not be collected immediately.
```

This is why explicit cleanup is still necessary for external resources.

---

## 490. What is deterministic resource cleanup?

It means releasing resources at a known time.

```js id="slklro"
const id = setInterval(work, 1000);

// Later
clearInterval(id);
```

For resources like file handles, sockets, observers, and timers, do not wait for GC.

---

## 491. What is RAII conceptually?

RAII means resource acquisition is tied to object lifetime. JavaScript does not traditionally use RAII heavily, but similar patterns can be built with explicit cleanup.

```js id="wx5zmr"
function withTimer(fn) {
  const start = performance.now();

  try {
    return fn();
  } finally {
    console.log(performance.now() - start);
  }
}
```

`finally` ensures cleanup logic runs.

---

## 492. What is `try...finally` used for?

It guarantees cleanup whether code succeeds or throws.

```js id="tjk2qd"
lock.acquire();

try {
  doWork();
} finally {
  lock.release();
}
```

This is useful for locks, timers, temporary state, and resource cleanup.

---

## 493. What is async cleanup with `finally`?

`finally` also works with `async/await`.

```js id="mfgi6b"
async function run() {
  const connection = await openConnection();

  try {
    return await connection.query("SELECT 1");
  } finally {
    await connection.close();
  }
}
```

The connection closes even if the query fails.

---

## 494. What is a resource leak?

A resource leak occurs when non-memory resources are not released.

```js id="ik7h4z"
function startSocket() {
  const socket = new WebSocket("wss://example.com");

  socket.addEventListener("message", event => {
    console.log(event.data);
  });

  return socket;
}
```

If the socket is no longer needed, call:

```js id="aa21so"
socket.close();
```

## 495. What is backoff for reconnecting WebSockets?

Reconnect attempts should wait increasingly longer.

```js id="2avw19"
function connect(attempt = 0) {
  const socket = new WebSocket("wss://example.com");

  socket.addEventListener("close", () => {
    const delay = Math.min(1000 * 2 ** attempt, 30000);

    setTimeout(() => {
      connect(attempt + 1);
    }, delay);
  });

  return socket;
}
```

Add jitter in production to avoid reconnect storms.

---

## 496. What is heartbeat for WebSocket health?

A heartbeat checks whether the connection is still alive.

```js id="j82mto"
function startHeartbeat(socket) {
  const id = setInterval(() => {
    if (socket.readyState === WebSocket.OPEN) {
      socket.send(JSON.stringify({ type: "ping" }));
    }
  }, 30000);

  return () => clearInterval(id);
}
```

Heartbeat timers also need cleanup.

## 497. What is Server-Sent Events?

Server-Sent Events allow a server to push text events to the browser over HTTP.

```js id="9jm1ed"
const source = new EventSource("/events");

source.addEventListener("message", event => {
  console.log(event.data);
});

// Later
source.close();
```

SSE is simpler than WebSocket for one-way server-to-client updates.

## 498. What is the difference between WebSocket and SSE?

WebSocket is bidirectional. SSE is server-to-client only.

```js id="lqr9ld"
// WebSocket
const socket = new WebSocket("wss://example.com");
socket.send("hello");
```

```js id="wud06x"
// SSE
const events = new EventSource("/events");
events.onmessage = event => console.log(event.data);
```

Use WebSocket for two-way real-time communication. Use SSE for simpler live updates.

## 499. What is polling?

Polling repeatedly asks the server for updates.

```js id="esdwst"
const id = setInterval(async () => {
  const data = await fetch("/updates").then(r => r.json());
  render(data);
}, 5000);
```

Polling is simple but can waste network and battery. Always clear the interval when no longer needed.

## 500. What should a senior JavaScript developer understand beyond syntax?

A senior developer understands execution, memory, async behavior, browser/runtime APIs, testing, security, architecture, performance, and failure modes.

```js id="fgrauw"
function createLiveSearch({ fetchJson, render, delay = 300 }) {
  let timerId;
  let requestId = 0;
  let cleanup = () => {};

  return function search(query) {
    clearTimeout(timerId);
    cleanup();

    timerId = setTimeout(async () => {
      const id = ++requestId;
      const controller = new AbortController();

      cleanup = () => controller.abort();

      try {
        const data = await fetchJson(`/search?q=${query}`, {
          signal: controller.signal
        });

        if (id === requestId) {
          render(data);
        }
      } catch (error) {
        if (error.name !== "AbortError") {
          throw error;
        }
      }
    }, delay);
  };
}
```

A beginner sees a debounced search.

A senior sees closure state, cancellation, stale response prevention, timer cleanup, dependency injection, async error handling, UI responsiveness, and testability.
