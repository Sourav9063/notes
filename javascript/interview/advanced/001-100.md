# Advanced JavaScript Questions & Answers with Code Examples

## 1. What is a closure?

A closure is created when a function retains access to variables from its outer lexical scope, even after that outer function has finished executing.

```js
function makeCounter() {
  let count = 0;

  return function increment() {
    count++;
    return count;
  };
}

const counter = makeCounter();

console.log(counter()); // 1
console.log(counter()); // 2
```

`increment` still has access to `count` because `count` is stored in the closure environment.

---

## 2. Do closures store values or references?

Closures capture **bindings**, not copied values. The inner function keeps access to the variable itself.

```js
function outer() {
  let value = 1;

  function inner() {
    console.log(value);
  }

  value = 2;
  return inner;
}

const fn = outer();
fn(); // 2
```

The closure does not remember `value` as `1`. It remembers the variable binding, whose latest value is `2`.

---

## 3. Why do closures sometimes cause memory leaks?

Closures can keep large objects alive longer than expected if the inner function remains reachable.

```js
function setup() {
  const hugeData = new Array(1_000_000).fill("data");

  return function handler() {
    console.log("clicked");
  };
}

const leakedHandler = setup();
```

Even though `handler` does not use `hugeData`, some engines may still keep the scope alive depending on optimization. A safer version avoids capturing unnecessary data.

```js
function setup() {
  const hugeData = new Array(1_000_000).fill("data");

  return function handler() {
    console.log("clicked");
  };
}

// Better: do not create large data in the same scope as long-lived closures.
```

---

## 4. What happens to closure variables during mark-and-sweep garbage collection?

If a closure is reachable, the variables in its lexical environment are also considered reachable.

```js
function createUserSession() {
  const user = { name: "Ava" };

  return function getUser() {
    return user;
  };
}

const session = createUserSession();
```

During mark-and-sweep, `session` is reachable, so the function object is marked. Its lexical environment is then marked, so `user` remains alive.

---

## 5. Why don’t cyclic references break modern garbage collectors?

Modern garbage collectors do not rely only on reference counts. They use reachability.

```js
let a = {};
let b = {};

a.other = b;
b.other = a;

a = null;
b = null;
```

Even though the two objects reference each other, they are unreachable from roots, so mark-and-sweep can collect them.

---

## 6. What are garbage collection roots?

Roots are objects that are directly reachable by the runtime, such as global variables, active stack frames, currently executing functions, and host references.

```js
let globalObj = { name: "rooted" };

function run() {
  const localObj = { name: "stack rooted" };
}

run();
```

`globalObj` remains reachable through the global object. `localObj` becomes unreachable after `run` finishes unless something keeps a reference to it.

---

## 7. What is the difference between lexical scope and dynamic scope?

JavaScript uses lexical scope. Variable lookup depends on where code is written, not where it is called.

```js
const value = "global";

function outer() {
  const value = "outer";

  return function inner() {
    console.log(value);
  };
}

const fn = outer();

function caller() {
  const value = "caller";
  fn();
}

caller(); // "outer"
```

The function uses the scope where it was created, not the scope where it was called.

---

## 8. What is the Temporal Dead Zone?

The Temporal Dead Zone is the period between entering a scope and the actual declaration initialization of `let` or `const`.

```js
console.log(x); // ReferenceError

let x = 10;
```

The variable exists in the scope but cannot be accessed before initialization.

---

## 9. Why does `var` behave differently from `let`?

`var` is function-scoped and initialized with `undefined`. `let` is block-scoped and remains uninitialized until its declaration is evaluated.

```js
console.log(a); // undefined
var a = 1;

console.log(b); // ReferenceError
let b = 2;
```

---

## 10. What is function hoisting?

Function declarations are hoisted with their full definition.

```js
sayHello(); // "Hello"

function sayHello() {
  console.log("Hello");
}
```

But function expressions assigned to `let` or `const` are not callable before initialization.

```js
sayHi(); // ReferenceError

const sayHi = function () {
  console.log("Hi");
};
```

---

## 11. What is the event loop?

The event loop is the mechanism used by the host environment to coordinate execution of tasks, microtasks, rendering, timers, I/O, and callbacks.

```js
console.log("A");

setTimeout(() => console.log("B"), 0);

console.log("C");
```

Output:

```js
A
C
B
```

The timer callback runs later, after the current call stack is empty.

---

## 12. Is the JavaScript event loop part of ECMAScript?

Not exactly. ECMAScript defines the language semantics, jobs, promises, execution contexts, and related behavior. The browser or Node.js host defines the event loop, timers, I/O, rendering, and task queues.

```js
setTimeout(() => {
  console.log("host API");
}, 0);
```

`setTimeout` is not part of core ECMAScript. It is provided by the host.

---

## 13. Why do Promise callbacks run before `setTimeout`?

Promise callbacks are microtasks. `setTimeout` callbacks are tasks/macrotasks. Microtasks are drained before the next task runs.

```js
setTimeout(() => console.log("timeout"), 0);

Promise.resolve().then(() => console.log("promise"));

console.log("sync");
```

Output:

```js
sync
promise
timeout
```

---

## 14. What are microtasks?

Microtasks are high-priority jobs that run after the current synchronous execution finishes but before the next task.

Common microtasks include:

```js
Promise.resolve().then(() => console.log("promise"));
queueMicrotask(() => console.log("microtask"));
```

Example:

```js
console.log("start");

queueMicrotask(() => console.log("microtask"));

console.log("end");
```

Output:

```js
start
end
microtask
```

---

## 15. What are macrotasks?

Macrotasks, often called tasks, include timers, events, network callbacks, and script execution.

```js
setTimeout(() => console.log("timer task"), 0);

console.log("sync");
```

Output:

```js
sync
timer task
```

---

## 16. What is event-loop starvation?

Event-loop starvation happens when microtasks or long synchronous code keep the event loop from processing other tasks such as rendering or user input.

```js
function starve() {
  Promise.resolve().then(starve);
}

starve();

setTimeout(() => {
  console.log("This may never run");
}, 0);
```

The microtask queue keeps refilling itself, preventing the timer from running.

---

## 17. How can you yield control to keep the UI responsive?

You can break long work into chunks and yield using `setTimeout`, `requestAnimationFrame`, or scheduler APIs where available.

```js
function processLargeArray(items) {
  let index = 0;

  function chunk() {
    const end = Math.min(index + 1000, items.length);

    while (index < end) {
      // process item
      index++;
    }

    if (index < items.length) {
      setTimeout(chunk, 0);
    }
  }

  chunk();
}
```

This lets the browser handle input and rendering between chunks.

## 18. What is the difference between `queueMicrotask` and `setTimeout`?

`queueMicrotask` runs before the next task. `setTimeout` schedules a future task.

```js
setTimeout(() => console.log("timeout"), 0);

queueMicrotask(() => console.log("microtask"));

console.log("sync");
```

Output:

```js
sync
microtask
timeout
```

## 19. What is the call stack?

The call stack tracks active function calls.

```js
function a() {
  b();
}

function b() {
  c();
}

function c() {
  console.log("inside c");
}

a();
```

When `c` runs, the stack is roughly:

```txt
c
b
a
global
```

---

## 20. What causes stack overflow?

Stack overflow happens when too many function calls are nested before previous calls return.

```js
function recurse() {
  recurse();
}

recurse(); // RangeError: Maximum call stack size exceeded
```

---

## 21. What is tail call optimization?

Tail call optimization allows a function call in tail position to reuse the current stack frame. Although specified in strict mode for some cases, it is not widely implemented in mainstream JavaScript engines.

```js
"use strict";

function factorial(n, acc = 1) {
  if (n <= 1) return acc;
  return factorial(n - 1, n * acc);
}
```

Even though this is tail-recursive, most engines will still grow the stack.

---

## 22. What is a hidden class?

A hidden class is an internal engine optimization that describes the shape of an object.

```js
const a = {};
a.x = 1;
a.y = 2;

const b = {};
b.x = 3;
b.y = 4;
```

Objects `a` and `b` likely share the same hidden class because properties were added in the same order.

---

## 23. How can property order affect performance?

Adding properties in different orders may create different hidden classes.

```js
const a = {};
a.x = 1;
a.y = 2;

const b = {};
b.y = 2;
b.x = 1;
```

Although both objects have `x` and `y`, engines may assign different hidden classes, reducing optimization opportunities.

---

## 24. What is inline caching?

Inline caching is an optimization where the engine remembers how a property was accessed previously.

```js
function getName(obj) {
  return obj.name;
}

getName({ name: "A" });
getName({ name: "B" });
getName({ name: "C" });
```

If objects have the same shape, the engine can optimize `obj.name` access.

---

## 25. What is monomorphic, polymorphic, and megamorphic access?

A call site is monomorphic if it sees one shape, polymorphic if it sees several, and megamorphic if it sees many.

```js
function readX(obj) {
  return obj.x;
}

readX({ x: 1 });
readX({ x: 2 }); // likely monomorphic

readX({ x: 3, y: 4 }); // another shape
readX({ x: 5, z: 6 }); // more shapes
```

Too many shapes can degrade optimization.

---

## 26. What is deoptimization?

Deoptimization happens when optimized code assumptions become invalid.

```js
function add(a, b) {
  return a + b;
}

add(1, 2);
add(3, 4);

// Engine may optimize for numbers.

add("hello", "world"); // assumption changes
```

The engine may discard optimized numeric code and fall back to generic behavior.

---

## 27. Why can mixing types hurt performance?

JavaScript engines specialize code based on observed types. Mixing types can invalidate assumptions.

```js
function double(x) {
  return x * 2;
}

double(10);
double(20);
double("30"); // type changes
```

The engine now needs to handle more generic cases.

---

## 28. What is JIT compilation?

JIT means Just-In-Time compilation. JavaScript engines interpret code first, collect runtime information, then compile hot code paths to optimized machine code.

```js
function sum(arr) {
  let total = 0;

  for (const n of arr) {
    total += n;
  }

  return total;
}

for (let i = 0; i < 10_000; i++) {
  sum([1, 2, 3, 4]);
}
```

Repeated execution may cause `sum` to become optimized.

---

## 29. Why are arrays with holes slower?

Sparse arrays require more complex lookup behavior.

```js
const dense = [1, 2, 3];

const sparse = [];
sparse[1000] = 1;
```

`dense` can be optimized as packed elements. `sparse` may need dictionary-like storage.

---

## 30. What is the difference between packed and holey arrays?

Packed arrays have continuous elements. Holey arrays have missing indices.

```js
const packed = [1, 2, 3];

const holey = [1, , 3];

console.log(0 in packed); // true
console.log(1 in holey);  // false
```

Holey arrays require extra checks during access.

---

## 31. Why can deleting object properties hurt performance?

Deleting properties can change an object’s internal representation.

```js
const user = {
  name: "Mia",
  age: 30
};

delete user.age;
```

Instead of deleting, setting to `undefined` may preserve shape in performance-sensitive code.

```js
user.age = undefined;
```

---

## 32. What is prototype inheritance?

Objects can delegate property lookup to another object through their prototype.

```js
const animal = {
  speak() {
    console.log("sound");
  }
};

const dog = Object.create(animal);
dog.name = "Rex";

dog.speak(); // "sound"
```

`dog` does not own `speak`; it finds it on its prototype.

---

## 33. What is the prototype chain?

The prototype chain is the sequence of objects searched when resolving a property.

```js
const grandparent = { a: 1 };
const parent = Object.create(grandparent);
parent.b = 2;

const child = Object.create(parent);
child.c = 3;

console.log(child.a); // 1
```

Lookup checks `child`, then `parent`, then `grandparent`.

---

## 34. What is the difference between `__proto__` and `prototype`?

`prototype` is a property on functions used when creating objects with `new`. `__proto__` refers to an object’s internal prototype.

```js
function Person(name) {
  this.name = name;
}

Person.prototype.sayHi = function () {
  console.log("Hi");
};

const p = new Person("Ava");

console.log(p.__proto__ === Person.prototype); // true
```

---

## 35. What happens when you use `new`?

`new` creates a new object, sets its prototype, binds `this`, runs the constructor, and returns the object unless the constructor returns another object.

```js
function User(name) {
  this.name = name;
}

const user = new User("Nora");

console.log(user.name); // "Nora"
```

Equivalent idea:

```js
const obj = {};
Object.setPrototypeOf(obj, User.prototype);
User.call(obj, "Nora");
```

---

## 36. What is `this` binding?

`this` depends on how a function is called, not where it is defined.

```js
const user = {
  name: "Ava",
  sayName() {
    console.log(this.name);
  }
};

user.sayName(); // "Ava"

const fn = user.sayName;
fn(); // undefined in strict mode
```

---

## 37. How do arrow functions handle `this`?

Arrow functions do not have their own `this`; they capture it lexically.

```js
const user = {
  name: "Ava",
  regular() {
    setTimeout(function () {
      console.log(this.name);
    }, 0);
  },
  arrow() {
    setTimeout(() => {
      console.log(this.name);
    }, 0);
  }
};

user.arrow(); // "Ava"
```

---

## 38. Why should methods usually not be arrow functions?

Arrow functions capture `this` from the surrounding scope, so they do not bind to the object as methods.

```js
const user = {
  name: "Ava",
  sayName: () => {
    console.log(this.name);
  }
};

user.sayName(); // likely undefined
```

Use method syntax instead:

```js
const user = {
  name: "Ava",
  sayName() {
    console.log(this.name);
  }
};
```

---

## 39. What is `call`, `apply`, and `bind`?

They control function `this`.

```js
function greet(greeting) {
  console.log(`${greeting}, ${this.name}`);
}

const user = { name: "Ava" };

greet.call(user, "Hello");      // Hello, Ava
greet.apply(user, ["Hi"]);      // Hi, Ava

const bound = greet.bind(user);
bound("Hey");                   // Hey, Ava
```

---

## 40. What is the difference between `Object.create` and class syntax?

`Object.create` directly creates an object with a given prototype. `class` is syntax over prototype-based inheritance.

```js
const animal = {
  speak() {
    console.log("sound");
  }
};

const dog = Object.create(animal);
dog.speak();
```

Class version:

```js
class Animal {
  speak() {
    console.log("sound");
  }
}

const dog = new Animal();
dog.speak();
```

---

## 41. Are JavaScript classes real classes?

JavaScript classes are syntactic sugar over prototypes, but with stricter semantics.

```js
class User {
  constructor(name) {
    this.name = name;
  }

  greet() {
    console.log(this.name);
  }
}

const user = new User("Ava");
```

`greet` is placed on `User.prototype`.

## 42. What are private class fields?

Private fields are declared with `#` and are enforced by the language.

```js
class BankAccount {
  #balance = 0;

  deposit(amount) {
    this.#balance += amount;
  }

  getBalance() {
    return this.#balance;
  }
}

const account = new BankAccount();
account.deposit(100);

console.log(account.getBalance()); // 100
// console.log(account.#balance);  // SyntaxError
```

---

## 43. How are private fields different from WeakMap privacy?

WeakMap privacy uses closure-held metadata. Native private fields are language-level and often more optimizable.

```js
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}

const user = new User("Ava");
console.log(user.getName());
```

WeakMap works, but private fields are cleaner:

```js
class User {
  #name;

  constructor(name) {
    this.#name = name;
  }

  getName() {
    return this.#name;
  }
}
```

---

## 44. Why use WeakMap?

WeakMap keys are weakly held. If the key object becomes unreachable elsewhere, it can be garbage collected.

```js
const metadata = new WeakMap();

let obj = {};
metadata.set(obj, { cached: true });

obj = null;
```

The metadata entry can be collected because the key is no longer reachable.

---

## 45. Why can’t WeakMap keys be primitives?

WeakMap keys must be objects because garbage collection reachability only applies to object references.

```js
const wm = new WeakMap();

wm.set({}, "ok");

// wm.set("name", "bad"); // TypeError
```

---

## 46. What is WeakSet?

WeakSet stores objects weakly. It is useful for tracking objects without preventing garbage collection.

```js
const seen = new WeakSet();

function process(obj) {
  if (seen.has(obj)) return;

  seen.add(obj);
  console.log("processing");
}

const item = {};
process(item);
process(item);
```

---

## 47. What is the difference between Map and Object?

`Map` supports any key type, preserves insertion order, and has better ergonomics for dynamic key-value storage.

```js
const map = new Map();

const key = { id: 1 };

map.set(key, "value");

console.log(map.get(key)); // "value"
```

Object keys are strings or symbols:

```js
const obj = {};
obj[key] = "value";

console.log(obj); // { "[object Object]": "value" }
```

---

## 48. What is the difference between Set and Array for membership checks?

`Set` is usually better for frequent membership checks.

```js
const ids = new Set([1, 2, 3]);

console.log(ids.has(2)); // true
```

Array requires scanning:

```js
const idsArray = [1, 2, 3];

console.log(idsArray.includes(2)); // true
```

## 49. What is object property descriptor?

Property descriptors define value, writability, enumerability, and configurability.

```js
const user = {};

Object.defineProperty(user, "id", {
  value: 123,
  writable: false,
  enumerable: true,
  configurable: false
});

user.id = 456;

console.log(user.id); // 123
```

---

## 50. What is the difference between enumerable and non-enumerable properties?

Enumerable properties show up in operations like `Object.keys` and `for...in`.

```js
const obj = {};

Object.defineProperty(obj, "hidden", {
  value: 42,
  enumerable: false
});

obj.visible = 1;

console.log(Object.keys(obj)); // ["visible"]
console.log(obj.hidden);       // 42
```

## 51. What is a getter?

A getter computes a property value when accessed.

```js
const user = {
  firstName: "Ava",
  lastName: "Stone",

  get fullName() {
    return `${this.firstName} ${this.lastName}`;
  }
};

console.log(user.fullName); // "Ava Stone"
```

---

## 52. What is a setter?

A setter runs when a property is assigned.

```js
const user = {
  firstName: "",
  lastName: "",

  set fullName(value) {
    const [first, last] = value.split(" ");
    this.firstName = first;
    this.lastName = last;
  }
};

user.fullName = "Ava Stone";

console.log(user.firstName); // "Ava"
```

## 53. What is a Proxy?

A Proxy lets you intercept operations on an object.

```js
const target = {
  name: "Ava"
};

const proxy = new Proxy(target, {
  get(obj, prop) {
    console.log(`Reading ${String(prop)}`);
    return obj[prop];
  }
});

console.log(proxy.name);
```

## 54. What is Reflect?

`Reflect` provides methods corresponding to internal object operations. It is often used inside proxies.

```js
const proxy = new Proxy(
  { name: "Ava" },
  {
    get(target, prop, receiver) {
      console.log(`get ${String(prop)}`);
      return Reflect.get(target, prop, receiver);
    }
  }
);

console.log(proxy.name);
```

## 55. What are symbols?

Symbols are unique primitive values often used as non-colliding property keys.

```js
const id = Symbol("id");

const user = {
  [id]: 123,
  name: "Ava"
};

console.log(user[id]); // 123
console.log(Object.keys(user)); // ["name"]
```

## 56. What is `Symbol.iterator`?

`Symbol.iterator` defines how an object is iterated.

```js
const range = {
  from: 1,
  to: 3,

  [Symbol.iterator]() {
    let current = this.from;
    const end = this.to;

    return {
      next() {
        if (current <= end) {
          return { value: current++, done: false };
        }

        return { done: true };
      }
    };
  }
};

for (const n of range) {
  console.log(n);
}
```

---

## 57. What is an iterator?

An iterator is an object with a `next()` method that returns `{ value, done }`.

```js
const iterator = {
  current: 0,

  next() {
    this.current++;

    if (this.current <= 3) {
      return { value: this.current, done: false };
    }

    return { done: true };
  }
};

console.log(iterator.next()); // { value: 1, done: false }
```

---

## 58. What is a generator?

A generator is a function that can pause and resume execution.

```js
function* numbers() {
  yield 1;
  yield 2;
  yield 3;
}

const gen = numbers();

console.log(gen.next()); // { value: 1, done: false }
console.log(gen.next()); // { value: 2, done: false }
```

---

## 59. How do generators help with lazy evaluation?

Generators produce values on demand.

```js
function* infiniteNumbers() {
  let n = 1;

  while (true) {
    yield n++;
  }
}

const nums = infiniteNumbers();

console.log(nums.next().value); // 1
console.log(nums.next().value); // 2
```

The infinite sequence does not allocate infinite memory.

---

## 60. What is an async function?

An async function always returns a Promise.

```js
async function getValue() {
  return 42;
}

getValue().then(console.log); // 42
```

This is equivalent to returning `Promise.resolve(42)`.

---

## 61. What does `await` do?

`await` pauses the async function until the promise settles, then resumes execution in a microtask.

```js
async function run() {
  console.log("A");

  await Promise.resolve();

  console.log("B");
}

run();

console.log("C");
```

Output:

```js
A
C
B
```

---

## 62. Does `await` block the main thread?

No. It pauses the async function, not the whole thread.

```js
async function wait() {
  await new Promise(resolve => setTimeout(resolve, 1000));
  console.log("done");
}

wait();

console.log("still runs immediately");
```

---

## 63. What is the difference between parallel and sequential async execution?

Sequential execution waits one operation after another.

```js
async function sequential() {
  const a = await fetch("/a");
  const b = await fetch("/b");

  return [a, b];
}
```

Parallel execution starts both before waiting.

```js
async function parallel() {
  const aPromise = fetch("/a");
  const bPromise = fetch("/b");

  const a = await aPromise;
  const b = await bPromise;

  return [a, b];
}
```

---

## 64. What is `Promise.all`?

`Promise.all` waits for all promises to fulfill, or rejects when one rejects.

```js
const results = await Promise.all([
  Promise.resolve(1),
  Promise.resolve(2),
  Promise.resolve(3)
]);

console.log(results); // [1, 2, 3]
```

---

## 65. What is `Promise.allSettled`?

`Promise.allSettled` waits for all promises regardless of success or failure.

```js
const results = await Promise.allSettled([
  Promise.resolve("ok"),
  Promise.reject("fail")
]);

console.log(results);
```

Output:

```js
[
  { status: "fulfilled", value: "ok" },
  { status: "rejected", reason: "fail" }
]
```

---

## 66. What is `Promise.race`?

`Promise.race` settles as soon as the first promise settles.

```js
const result = await Promise.race([
  new Promise(resolve => setTimeout(() => resolve("slow"), 1000)),
  new Promise(resolve => setTimeout(() => resolve("fast"), 100))
]);

console.log(result); // "fast"
```

---

## 67. What is `Promise.any`?

`Promise.any` fulfills with the first fulfilled promise. It rejects only if all promises reject.

```js
const result = await Promise.any([
  Promise.reject("bad"),
  Promise.resolve("good")
]);

console.log(result); // "good"
```

---

## 68. How do you cancel async work?

Promises themselves are not cancellable, but APIs can support cancellation through `AbortController`.

```js
const controller = new AbortController();

fetch("/data", {
  signal: controller.signal
}).catch(error => {
  if (error.name === "AbortError") {
    console.log("Request cancelled");
  }
});

controller.abort();
```

---

## 69. What is a race condition in JavaScript?

A race condition happens when the final result depends on unpredictable timing.

```js
let state = "";

async function loadA() {
  await new Promise(resolve => setTimeout(resolve, 100));
  state = "A";
}

async function loadB() {
  await new Promise(resolve => setTimeout(resolve, 50));
  state = "B";
}

loadA();
loadB();
```

`state` may end as `"A"` even though `loadB` finished first.

---

## 70. How do you prevent stale async results?

Use a request ID or cancellation.

```js
let latestRequestId = 0;

async function search(query) {
  const requestId = ++latestRequestId;

  const result = await fetch(`/search?q=${query}`);

  if (requestId !== latestRequestId) {
    return;
  }

  console.log("Use result", result);
}
```

Only the latest request updates the UI.

---

## 71. What is `structuredClone`?

`structuredClone` creates a deep clone using the structured clone algorithm.

```js
const original = {
  name: "Ava",
  date: new Date(),
  nested: { x: 1 }
};

const copy = structuredClone(original);

console.log(copy.date instanceof Date); // true
console.log(copy.nested === original.nested); // false
```

---

## 72. Why is `JSON.parse(JSON.stringify(obj))` not a safe deep clone?

It loses unsupported values and types.

```js
const original = {
  date: new Date(),
  undefinedValue: undefined,
  fn: () => {},
  infinity: Infinity
};

const copy = JSON.parse(JSON.stringify(original));

console.log(copy);
```

The date becomes a string, `undefined` and functions disappear, and `Infinity` becomes `null`.

---

## 73. Can `structuredClone` clone functions?

No. Functions are not cloneable.

```js
const obj = {
  fn() {}
};

// structuredClone(obj); // DataCloneError
```

---

## 74. What are transferables in `structuredClone`?

Transferables move ownership instead of copying.

```js
const buffer = new ArrayBuffer(8);

const cloned = structuredClone(buffer, {
  transfer: [buffer]
});

console.log(buffer.byteLength); // 0
console.log(cloned.byteLength); // 8
```

The original buffer is detached.

## 75. What is ESM?

ESM stands for ECMAScript Modules. It uses static `import` and `export`.

```js
// math.js
export function add(a, b) {
  return a + b;
}

// app.js
import { add } from "./math.js";

console.log(add(1, 2));
```

---

## 76. Why are ESM imports static?

Static imports allow the engine to analyze module dependencies before execution.

```js
import { add } from "./math.js";
```

This helps with linking, tree-shaking, and detecting errors earlier.

Dynamic import is available when runtime loading is needed:

```js
const module = await import("./math.js");
console.log(module.add(1, 2));
```

---

## 77. What are live bindings in ESM?

ESM imports are live views of exported bindings, not copied values.

```js
// counter.js
export let count = 0;

export function increment() {
  count++;
}

// app.js
import { count, increment } from "./counter.js";

console.log(count); // 0
increment();
console.log(count); // 1
```

The imported `count` reflects updates.

---

## 78. What happens in circular ESM imports?

ESM supports cycles through live bindings, but accessing uninitialized bindings can cause errors.

```js
// a.js
import { b } from "./b.js";

export const a = "A";

console.log(b);

// b.js
import { a } from "./a.js";

export const b = "B";

console.log(a);
```

Depending on evaluation order, one module may access a binding before it is initialized.

---

## 79. What is top-level `await`?

Top-level `await` allows modules to wait during evaluation.

```js
// config.js
export const config = await fetch("/config").then(r => r.json());
```

Any module importing this one waits for it to finish evaluating.

---

## 80. How does CommonJS differ from ESM?

CommonJS is runtime-based and uses `require`. ESM is statically analyzable and uses `import`.

```js
// CommonJS
const fs = require("fs");

module.exports = {
  name: "demo"
};
```

ESM:

```js
import fs from "fs";

export const name = "demo";
```

CommonJS exports are object-based; ESM exports are live bindings.

---

## 81. What is execution context?

An execution context contains information needed to run code, including lexical environment, variable environment, and `this` binding.

```js
const globalValue = 1;

function run() {
  const localValue = 2;
  console.log(globalValue + localValue);
}

run();
```

Calling `run` creates a new function execution context.

---

## 82. What is lexical environment?

A lexical environment stores identifier bindings and a reference to an outer environment.

```js
function outer() {
  const x = 1;

  function inner() {
    const y = 2;
    console.log(x + y);
  }

  inner();
}
```

`inner` has its own environment and a reference to `outer`’s environment.

---

## 83. What is variable environment?

Historically, the variable environment tracks `var` and function declarations, while the lexical environment tracks `let`, `const`, and class declarations.

```js
function test() {
  console.log(a); // undefined
  var a = 1;

  // console.log(b); // ReferenceError
  let b = 2;
}
```

---

## 84. What is the difference between `==` and `===`?

`===` checks strict equality without coercion. `==` allows coercion.

```js
console.log(1 == "1");  // true
console.log(1 === "1"); // false
```

Prefer `===` unless coercion is intentional.

---

## 85. Why is `NaN !== NaN`?

`NaN` represents an invalid number result and is not equal to itself.

```js
console.log(NaN === NaN); // false

console.log(Number.isNaN(NaN)); // true
```

Use `Number.isNaN`.

---

## 86. What is `Object.is`?

`Object.is` is similar to `===`, but treats `NaN` as equal to itself and distinguishes `+0` from `-0`.

```js
console.log(Object.is(NaN, NaN)); // true
console.log(Object.is(+0, -0));   // false

console.log(+0 === -0);           // true
```

---

## 87. What is type coercion?

Type coercion is automatic conversion between types.

```js
console.log("5" - 1); // 4
console.log("5" + 1); // "51"
```

`-` coerces to numbers. `+` performs string concatenation if either operand is a string.

---

## 88. What is truthy and falsy?

Falsy values are values treated as false in boolean contexts.

```js
const values = [false, 0, -0, 0n, "", null, undefined, NaN];

for (const value of values) {
  console.log(Boolean(value));
}
```

Everything else is truthy.

---

## 89. What is optional chaining?

Optional chaining safely accesses nested properties.

```js
const user = {};

console.log(user.profile?.address?.city); // undefined
```

Without it:

```js
// user.profile.address.city
// TypeError if profile is undefined
```

---

## 90. What is nullish coalescing?

`??` uses the right-hand value only when the left-hand value is `null` or `undefined`.

```js
const count = 0;

console.log(count || 10); // 10
console.log(count ?? 10); // 0
```

Use `??` when `0`, `false`, or `""` are valid values.

---

## 91. What is destructuring?

Destructuring extracts values from arrays or objects.

```js
const user = {
  name: "Ava",
  age: 30
};

const { name, age } = user;

console.log(name, age);
```

Array destructuring:

```js
const [first, second] = [10, 20];

console.log(first); // 10
```

---

## 92. What is rest syntax?

Rest collects remaining values.

```js
const [first, ...rest] = [1, 2, 3, 4];

console.log(first); // 1
console.log(rest);  // [2, 3, 4]
```

Object rest:

```js
const { password, ...safeUser } = {
  name: "Ava",
  password: "secret"
};

console.log(safeUser); // { name: "Ava" }
```

---

## 93. What is spread syntax?

Spread expands values.

```js
const a = [1, 2];
const b = [3, 4];

const combined = [...a, ...b];

console.log(combined); // [1, 2, 3, 4]
```

Object spread:

```js
const user = { name: "Ava" };
const updated = { ...user, active: true };

console.log(updated);
```

---

## 94. Is object spread a deep copy?

No. Object spread performs a shallow copy.

```js
const original = {
  nested: {
    count: 1
  }
};

const copy = { ...original };

copy.nested.count = 2;

console.log(original.nested.count); // 2
```

The nested object is shared.

---

## 95. What is immutability?

Immutability means not changing existing data directly, but creating updated copies.

```js
const user = {
  name: "Ava",
  age: 30
};

const updatedUser = {
  ...user,
  age: 31
};

console.log(user.age);        // 30
console.log(updatedUser.age); // 31
```

This helps with predictable state updates.

---

## 96. What is `Object.freeze`?

`Object.freeze` prevents adding, removing, or changing own properties.

```js
const user = Object.freeze({
  name: "Ava"
});

user.name = "Mia";

console.log(user.name); // "Ava"
```

It is shallow:

```js
const obj = Object.freeze({
  nested: { count: 1 }
});

obj.nested.count = 2;

console.log(obj.nested.count); // 2
```

---

## 97. What is memory retention?

Memory retention happens when objects remain reachable even though they are no longer needed.

```js
const cache = new Map();

function loadUser(id) {
  const user = { id, data: new Array(100000).fill("x") };
  cache.set(id, user);
}

loadUser(1);
```

If the cache never evicts entries, memory grows indefinitely.

---

## 98. How can event listeners cause memory leaks?

If an event listener references an object, and the listener is not removed, the object may stay alive.

```js
function mount() {
  const data = new Array(100000).fill("x");

  function onClick() {
    console.log(data.length);
  }

  window.addEventListener("click", onClick);

  return function unmount() {
    window.removeEventListener("click", onClick);
  };
}

const cleanup = mount();
cleanup();
```

Always clean up long-lived listeners when components or objects are destroyed.

---

## 99. How can closures and timers cause memory leaks?

A timer callback can keep closed-over variables alive.

```js
function startPolling() {
  const largeData = new Array(100000).fill("x");

  const id = setInterval(() => {
    console.log(largeData.length);
  }, 1000);

  return () => clearInterval(id);
}

const stop = startPolling();

// Later:
stop();
```

If `clearInterval` is never called, `largeData` remains reachable.

## 100. What separates basic JavaScript knowledge from engine-level JavaScript understanding?

Basic JavaScript knowledge explains what code does. Engine-level understanding explains **why**, including scope representation, execution contexts, event-loop scheduling, garbage collection, hidden classes, inline caching, JIT assumptions, module linking, and host/runtime behavior.

Example:

```js
console.log("A");

setTimeout(() => console.log("B"), 0);

Promise.resolve().then(() => {
  console.log("C");
});

queueMicrotask(() => {
  console.log("D");
});

console.log("E");
```

Output:

```js
A
E
C
D
B
```

A basic answer says: “Promises run before timers.”

A senior answer says: synchronous code runs first, Promise reactions and `queueMicrotask` callbacks are microtasks, microtasks are drained before the event loop takes the next task, and timer callbacks are scheduled through the host environment.

That is the difference between using JavaScript and understanding the runtime.
